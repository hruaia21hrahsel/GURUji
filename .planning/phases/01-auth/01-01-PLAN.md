**lib/supabase.ts:** Replace existing lazy singleton with a proper Supabase client:
- Import `createClient` from `@supabase/supabase-js`
- Import `LargeSecureStore` from `./large-secure-store`
- Read `EXPO_PUBLIC_SUPABASE_URL` and `EXPO_PUBLIC_SUPABASE_ANON_KEY` from `expo-constants` or `process.env`
- Create client with `auth: { storage: new LargeSecureStore(), autoRefreshToken: true, persistSession: true, detectSessionInUrl: false }`
- Export `supabase` singleton
- If env vars are missing, log a warning but don't crash (development convenience)

**lib/auth.ts:** Auth helper functions:
- `syncProfileToSupabase(userId: string)` — reads class/board/language from useAppStore, upserts to `profiles` table (as shown in Research code example for AUTH-05)
- `signOut()` — calls `supabase.auth.signOut()`, then `useAppStore.getState().clearAuth()`, then `usePinStore.getState().reset()`

**store/usePinStore.ts:** Zustand store (NO persist — PIN state is derived from SecureStore on app launch):
- State: `isLocked` (boolean, default true), `hasPin` (boolean, default false), `failedAttempts` (number, default 0), `biometricEnabled` (boolean, default false)
- Actions: `unlock()`, `lock()`, `incrementFailedAttempts()`, `resetFailedAttempts()`, `setHasPin(boolean)`, `setBiometricEnabled(boolean)`, `reset()`
- `isLocked` starts as `true` — set to `false` only after successful PIN/biometric entry
- `failedAttempts` resets after successful unlock
- After `failedAttempts >= 3`, the PIN screen must show "Too many attempts — sign in again" and trigger full re-auth
