**lib/pin.ts:** PIN utility functions:
1. `hashPin(pin: string): string` — SHA-256 hash of PIN + static app salt. Use a simple deterministic hash. Import crypto or use a lightweight hash (since we have `react-native-get-random-values`, we can use `crypto.subtle.digest` or a simple custom hash). For v1, store the PIN directly in SecureStore (it's only 4 digits, and SecureStore is already hardware-backed — hashing adds complexity without security benefit since the attacker would need SecureStore access). Decision: store PIN plaintext in SecureStore per research recommendation ("4-digit hash with a salt, or simply store in SecureStore directly given its small size").
2. `savePin(pin: string): Promise<void>` — Stores PIN in SecureStore under key `guruji_pin`. Updates `usePinStore.getState().setHasPin(true)`.
3. `getPin(): Promise<string | null>` — Reads from SecureStore key `guruji_pin`.
4. `verifyPin(input: string): Promise<boolean>` — Compares input with stored PIN. Returns true if match.
5. `checkBiometricAvailability(): Promise<boolean>` — Calls `LocalAuthentication.hasHardwareAsync()` AND `LocalAuthentication.isEnrolledAsync()`. Returns true only if both are true.
6. `authenticateWithBiometric(): Promise<boolean>` — Calls `LocalAuthentication.authenticateAsync({ promptMessage: 'Unlock GURUji', fallbackLabel: 'Use PIN', cancelLabel: 'Cancel' })`. Returns `result.success`.

**app/auth/pin-setup.tsx — PIN Setup Screen:**
Shown after first successful authentication (when user has no PIN yet).
- Title: "Set your PIN" with subtitle "You'll use this every time you open GURUji"
- 4 input circles (filled/empty visual indicators) — NOT text inputs, use a custom numeric keypad
- Custom numeric keypad (1-9, 0, backspace) at bottom of screen — large touch targets for kids
- Flow: Enter 4 digits -> "Confirm your PIN" -> enter same 4 digits again
- If confirmation doesn't match: shake animation (or red flash), "PINs don't match — try again", reset both
- On success: call `savePin(pin)`, then offer biometric: "Enable fingerprint unlock?" with Yes/No buttons. If yes, call `checkBiometricAvailability()` — if available, call `authenticateWithBiometric()` to confirm, then `usePinStore.getState().setBiometricEnabled(true)`. If biometric not available, show "Biometric not available on this device" and skip.
- After PIN saved + biometric choice: call `usePinStore.getState().unlock()` and navigate to tabs (the first time, user goes straight in after setting PIN)

**app/pin/index.tsx — PIN Entry Screen:**
Shown every time app opens when user `isAuthenticated && isLocked`.
- "Welcome back!" header with GURUji wordmark
- 4 circles for PIN display (same visual as setup)
- Custom numeric keypad (same component/style as setup)
- On mount: if `biometricEnabled`, auto-trigger `authenticateWithBiometric()`. If success, call `usePinStore.getState().unlock()`.
- On 4-digit entry: call `verifyPin(input)`. If correct, `unlock()`. If wrong, `incrementFailedAttempts()`, shake animation, show "Wrong PIN" with attempts remaining (3 - failedAttempts).
- When `failedAttempts >= 3`: hide PIN pad, show "Too many attempts" message and "Sign in again" button. Button calls `signOut()` which clears auth + PIN state and redirects to auth screen.
- "Forgot PIN?" link at bottom — calls `signOut()` and redirects to auth screen (user re-authenticates, then sets new PIN). Per user decision, PIN reset sends OTP to phone — but for v1, signing out and re-authing achieves the same result.
